# ジム向け管理Webアプリ（学び直し用）

## このリポジトリについて
本リポジトリは、チームで開発したジム向け管理Webアプリを  
**学び直し・振り返り目的でForkした個人用リポジトリ**です。

元プロジェクトでは、実務を想定したWebアプリ開発を通じて、
要件整理・DB設計・CRUD実装・チーム開発の流れを経験しました。

本リポジトリでは、当時のコード・設計資料を見直しながら、
「なぜその設計・実装になったのか」「今ならどう改善するか」
を言語化することを目的としています。

---

## 開発背景・目的
- チーム開発の流れを経験するため
- CRUDを中心としたバックエンド処理の理解
- DB設計がアプリ全体に与える影響を学ぶため
- 実務に近い構成でWebアプリを作る経験を積むため

---

## アプリの概要
ジム管理者が利用することを想定したWebアプリで、主に以下の機能を持っています。

- 会員管理
- トレーニングメニュー管理
- トレーニング記録（セット・回数・重量など）の登録
- 管理者向けの基本的なCRUD機能

---

## 技術スタック
- Backend：Laravel（PHP）
- Frontend：Blade / Livewire
- Database：MySQL
- その他：Tailwind CSS / GitHub

---

## チーム開発について
- チーム人数：［例：3名］
- 役割分担を行いながら開発
- GitHubを用いたソースコード管理・共同開発

---

## 自分の担当範囲
- データベース設計（migration / seeder）
- バックエンドのCRUD処理の実装
- 不具合修正・既存機能の調整

※ フロントエンドを含め、すべてを単独で担当したわけではありません。

---

## 機能要件・設計時に意識していたこと（要約）
### 機能面
- 管理者が直感的に操作できる画面構成
- CRUD処理を明確に分け、処理の流れを追いやすくする
- 将来的な機能追加を想定した構成

### DB設計面
- users を起点に各テーブルを関連付ける構成
- トレーニングメニューと実績データを分離
- 正規化を意識し、冗長なデータを持たせない設計

---

## 学び直して分かったこと
- DB設計の良し悪しが、実装・保守性に大きく影響すること
- Laravelにおける routing / controller / model の役割の整理
- チーム開発では「設計意図の共有」が非常に重要であること
- 後からコードを読む人の視点を意識する必要性

---

## 設計と実装のズレ
- 初期の機能要件から、実装段階で簡略化・変更した部分がある
- 開発スピードを優先し、設計を詰め切れなかった箇所が存在する
- 当時は動作優先で、改善余地を残した実装もあった

---

- バリデーション・テストの追加
- 責務分離をより明確にした設計
- チーム内で設計意図をドキュメント化する
